# Техническое описание Telegram-бота для управления VPN

## Стек технологий
- **Язык программирования:** Python
- **База данных:** SQLite
- **Библиотеки:** 
  - `aiogram` 3 — взаимодействие с Telegram
  - `aiohttp` — асинхронные запросы к VPN API
  - `qrcode` + `pillow` — генерация QR-кодов для ключей

### Механизмы защиты от race conditions
- **FSM (Finite State Machine)**: Управление многошаговыми диалогами и состояниями пользователей
  - Хранение контекста диалога (выбранный тариф, данные регистрации)
  - Маршрутизация обработчиков в зависимости от состояния
  - Валидация последовательности действий пользователя
  
- **asyncio.Lock**: Атомарная защита критичных операций от дублирования
  - Блокировка на уровне user_id для предотвращения одновременных запросов
  - Применяется для:
    - Создания VPN-ключей (защита от двойного создания при быстрых кликах)
    - Обработки платежей (предотвращение двойных списаний)
    - Операций с балансом пользователя
    - Любых долгих API-вызовов к VPN-серверу

## Описание проекта
Telegram-бот для управления VPN-сервером через API. И продажа доступа клиентам через https://t.me/Ya_SellerBot за крипту и напрямую за звезды.

### Основные возможности
1. **Универсальная поддержка протоколов:**
   - **VLESS**: Reality (xtls-rprx-vision), WS, TCP, gRPC.
   - **VMess**: WS, TCP, gRPC.
   - **Trojan**: TCP, WS, gRPC.
   - **Shadowsocks**: TCP.
   - Автоматическое определение настроек inbound (transport, security, flow).

2. **Управление пользователями VPN:**
   - Добавление новых пользователей.
   - Выдача ключей доступа (автогенерация ссылок и QR).
   - Взаимодействие с VPN-сервером через API.
   - Прием оплаты криптой через https://yadreno.ru/seller/integration.php (это документация по интеграции). Интеграцию мы делаем только через колбек ссылки, без вебхуков
   - Прием платежей через Telegram Stars.
   - Прием платежей с банковских карт (в рублях) через ЮКассу (Telegram Payments API).

3. **Пробная подписка:**
   - Возможность выдать пользователю бесплатный тестовый период (1 раз на аккаунт).
   - Флаг использования хранится в таблице пользователей (`used_trial`).
   - Реализуется через создание реального заказа со статусом `paid` и типом `trial`.
   - Настраивается админом: вкл/выкл, текст страницы, выбор тарифа.
   - Активируется кнопкой на главной странце (если доступна и не использована).

## Архитектура и Структура проекта (Гайд для Разработчика)

Этот раздел предназначен для быстрого погружения в контекст проекта.

### 📂 Карта файловой системы
*   `main.py`: **Точка входа**. Инициализирует бота, диспетчер и логгер. Запускает полинг.
*   `config.py`: **Конфигурация**. Хранит `BOT_TOKEN`, `ADMIN_IDS`, константы для rate limiting и retry. **Важно**: VPN-серверы и тарифы хранятся в БД и управляются через админ-панель.
*   `bot/`: Основная директория бота.
    *   `handlers/`: **Роутеры и Хендлеры**. Логика разделена на зоны:
        *   `admin/`: Админ-панель (управление юзерами, серверами, статистикой).
        *   `user/`: Пользовательская зона (покупка, инструкции, профиль).
    *   `services/`: **Бизнес-логика**. Код, не зависящий от Telegram.
        *   `vpn_api.py`: Взаимодействие с API VPN-панелей (создание/удаление ключей).
        *   `billing.py`: Логика проверки оплат и начисления баланса.
    *   `states/`: **FSM**. Классы состояний (`StatesGroup`) для сценарных диалогов.
    *   `middlewares/`: Прослойки (проверка админов, логирование).
    *   `keyboards/`: Генераторы клавиатур.
*   `database/`: Работа с данными.
    *   `requests.py`: **Единственная точка доступа к БД**. Хендлеры вызывают функции отсюда. Прямой SQL в хендлерах запрещен.

### 📐 Принципы разработки
1.  **Разделение ответственности**: Хендлер (`handlers`) парсит ввод и вызывает сервис (`services`) или БД (`database`). Хендлер не должен содержать сложной бизнес-логики.
2.  **Модульность**: Новые фичи добавляем в соответствующие подпапки. Если фича крупная — создаем под нее отдельный пакет в `handlers`.
3.  **Конфигурация**: Никаких хардкод-значений (токенов, паролей) в коде. Всё берем из `config.py`.
4.  Все тексты сообщений форматируем используя стандартный Markdown (Legacy): жирный через `*text*`, курсив через `_text_`. Не используем `**` или `MarkdownV2` если не требуется экранирование спецсимволов.

### 🗄️ Правила работы с базой данных SQLite

2.  **Библиотека**: Используем встроенную библиотеку `sqlite3` (не требует установки)
3.  **Инициализация**: Создаем функцию `init_db()` в `database/connection.py` для создания таблиц при первом запуске
4.  **Безопасность**: Всегда используем параметризированные запросы (знак `?`) для защиты от SQL-инъекций
5.  **Транзакции и соединения**: 
    - Используем **контекстный менеджер** `with get_db() as conn:` для автоматического commit/rollback
    - Используем `row_factory = sqlite3.Row` для доступа к полям по имени
    - Транзакции обеспечивают атомарность операций (например, списание баланса + создание VPN ключа)
    - Контекстный менеджер автоматически делает `commit()` при успехе и `rollback()` при ошибке
6.  **Структура**:
    - `database/connection.py` - функции для работы с соединением и инициализации БД
    - `database/requests.py` - все функции для работы с данными (единственная точка доступа к БД)
    - `database/migrations.py` - система миграций БД (см. [migrations.py](database/migrations.py))
7.  **Миграции БД**:
    - Все изменения структуры БД **только через миграции** в файле `database/migrations.py`
    - Миграции проверяются и применяются автоматически при действиях администратора
    - **Актуальная структура БД** описана в функциях миграций в файле [migrations.py](database/migrations.py)
    - **Как добавить новую миграцию:**
        1. Создайте функцию `migration_N()` где N - следующий номер версии
        2. Напишите SQL команды для изменения структуры
        3. Добавьте функцию в словарь `MIGRATIONS`
        4. Обновите константу `LATEST_VERSION`

## 🎯 Критически важное

### 1. Логирование и мониторинг
- **Система логирования**: Используем встроенный модуль `logging` с настройкой уровней
  - `DEBUG`: Для разработки (детальная информация о работе бота)
  - `INFO`: Для продакшн (ключевые события: создание ключей, платежи, команды)
  - `WARNING/ERROR`: Ошибки API, неудачные платежи, исключения
- **Что логируем обязательно**:
  - Все запросы к VPN API (с результатом и временем выполнения)
  - Платежные операции (сумма, user_id, статус)
  - Критичные операции (создание/удаление ключей, изменение баланса)
  - Ошибки и исключения с полным traceback
- **Формат логов**: `[timestamp] [level] [user_id] [action] - message`
- **Файлы логов**: 
  - `logs/bot.log` - основной лог
  - `logs/payments.log` - отдельный лог платежей
  - Ротация логов при достижении 10MB
- **Статус БД при запуске**: Бот логирует текущую версию схемы БД и статус применения миграций (INFO).

### 2. Обработка ошибок
- **Retry для внешних API**:
  - Повторяем неудачные запросы с экспоненциальной задержкой
  - Максимум 3 попытки для VPN API
  - Интервалы: 1 сек → 3 сек → 9 сек
- **Уведомления администратора**:
  - При критичных ошибках (падение VPN API, ошибки БД)
  - Отправка в Telegram админам из `ADMIN_IDS`
- **Graceful degradation**:
  - Если VPN API недоступен → сохраняем запрос в очередь БД для повтора
  - Пользователь получает сообщение "Попробуйте через минуту"
- **Глобальный обработчик исключений**:
  - Middleware для перехвата всех необработанных ошибок
  - Логирование + уведомление админа + ответ пользователю

### 3. Валидация входных данных
- **Суммы платежей**: Только положительные числа, диапазон 1-10000
- **Callback data**: Валидация структуры (например, `action:server_id:user_id`)
- **User input**: Защита от SQL-инъекций через параметризированные запросы
- **Длина текста**: Ограничение на длину вводимых данных (имя, описание)

### 4. Graceful Shutdown
- **Корректное завершение работы** при получении сигналов SIGINT/SIGTERM:
  - Завершение обработки текущих запросов (не обрывать на середине)
  - Сохранение FSM-состояний пользователей в БД
  - Закрытие всех соединений с БД
  - Закрытие сессий aiohttp для VPN API
- **Реализация**: Обработка сигналов через `signal.signal()` с корректным `await bot.session.close()`

## 🛡️ Безопасность

### Rate Limiting (защита от спама)
- **Middleware для ограничения частоты запросов**:
  - Обычные пользователи: максимум **30 команд в минуту**
  - Критичные операции (создание ключа, платеж): **5 операций в минуту**
  - Администраторы: без ограничений
- **Реализация**: 
  - Хранение счетчиков в памяти (dict с user_id и timestamp)
  - Очистка старых записей каждые 5 минут
- **Ответ при превышении**: "⏳ Слишком много запросов, подождите немного"

### Идемпотентность операций
- **Защита от дубликатов**:
  - Проверка существующих активных ключей перед созданием нового
  - UUID для каждой транзакции платежа (повторная проверка не создает дубликат)
  - Уникальные индексы в БД для предотвращения дублей на уровне данных

### Стиль и принцип создания диалогов
-  только инлайн кнопки
-  каждая кнопка начинается с эмодзи, который отражает ее суть
-  каждое сообщение имеет заголовок: тематический эмодзи, и жирным шрифтом суть этого сообщения(страницы), после заголовка пустая строка
-  тексты пишем веселые, дружеские и с юмором, особенно для администратора
-  Весь интерфейс бота должен быть на русском языке
-  **Форматирование цен**: Все цены в боте выводятся без лишних дробных нулей. Целые числа пишутся без запятой (`1` вместо `1.0`), дробные пишутся лаконично (`1.5` вместо `1.50`). Для этого всегда используется функция `format_price_compact`.

### Правила навигации
-  **Кнопка «🈴 На главную»** — всегда располагается внизу **отдельным рядом** (одна большая кнопка на всю ширину)
-  Если на экране есть и **«⬅️ Назад»** и **«🈴 На главную»** — они располагаются в **последнем ряду вместе** (две кнопки в ряд)
-  На **главном экране** кнопка «На главную» не отображается

- Описание преимуществ (редактируется админом)

Экран после нажатия на конкретный ключ:

  - `[📈 Продлить]` — продление подписки
  - `[🗑 Удалить]` — удаление ключа из БД и 3X-UI панели
- **Безопасность**: строгие проверки на принадлежность ключа (любые действия с ключом возможны только в случае, если Telegram ID нажавшего совпадает с `user_id` владельца ключа в БД).
- Последний ряд: `[⬅️ Назад]` `[🈴 На главную]`

## ⚙️ Структура админ-панели

Доступ к админ-панели имеют только пользователи из списка `ADMIN_IDS` в `config.py`.

### Точка входа
- При `/start` администратор видит дополнительную кнопку **«⚙️ Админ-панель»**
- Проверка: `telegram_id in ADMIN_IDS`

### 📊 Главный экран админки
Показывает сводную статистику по всем серверам:
- Для каждого сервера: 🟢/🔴 статус, активные клиенты, трафик, CPU
- Кнопки:
  - `[🖥️ Сервера]`
  - `[💳 Оплаты]`
  - `[📢 Рассылка]`
  - `[👥 Пользователи]`
  - `[⚙️ Настройки бота]`
  - `[🎁 Пробная подписка]`
  - `[📥 Скачать логи]`
  - `[🈴 На главную]`

### 📥 Скачать логи
Раздел для загрузки файлов логов. Сразу после нажатия кнопки пользователю отправляется меню с выбором логов для скачивания.
- Кнопки:
  - Первый ряд: `[📄 Полный лог]` `[⚠️ Ошибки]`
  - Второй ряд: `[⬅️ Назад]` `[🈴 На главную]`

### ⚙️ Настройки бота
Раздел для управления ботом и автообновления с GitHub.

#### Главный экран
- Информация о текущей версии (коммит, ветка)
- Статус подключения к GitHub
- Кнопки:
  - `[🔍 Проверить обновления]` — проверяет наличие новых коммитов
  - `[🔄 Обновить бота]` — выполняет `git pull` и перезапуск
  - `[✏️ Изменить тексты]` — редактирование текстов: Главная, Справка, Ссылки (Новости/Поддержка)
  - `[🛑 Остановить бота]` — остановка бота с подтверждением
  - `[⬅️ Назад]` `[🈴 На главную]`

#### Автообновление с GitHub
- **Настройка**: URL репозитория указывается в `config.py` → `GITHUB_REPO_URL`
- **Проверка обновлений**: выполняет `git fetch` и показывает количество новых коммитов
- **Обновление**: выполняет `git pull`, показывает лог изменений и перезапускает бота
- **Автоматическая проверка**: Ежедневно в 12:00 бот проверяет обновления и уведомляет администраторов.
- **Защита**: проверка на локальные изменения, обработка конфликтов

## 🔌 Особенности работы с API 3X-UI

При разработке опираемся на описанное АПИ по ссылке https://documenter.getpostman.com/view/5146551/2sB3QCTuB6#intro
В ходе разработки и реверс-инжиниринга API панели 3X-UI были выявлены следующие критические особенности:

### 1. Аутентификация
- Используется **Session-based** аутентификация (Cookie), а не JWT или API токены.
- После успешного логина (`POST /login`) сервер устанавливает куку `3x-ui`.
- **Важно**: Клиент должен поддерживать сохранение и отправку кук (Cookie Jar).
- Для хостов с IP-адресами (без домена) необходимо использовать `aiohttp.CookieJar(unsafe=True)`, иначе куки игнорируются.

### 2. Структура URL
- Все API запросы должны включать **секретный путь** (webBasePath), настроенный в панели.
- Формат: `https://{host}:{port}/{webBasePath}/panel/api/inbounds/...`
- Пример: `https://144.124.255.92:2053/5949935108817e64fa5e63b0e0aba8e3/panel/api/inbounds/list`

### 3. Обязательные заголовки (Headers)
```python
headers = {
    "Accept": "application/json",
    "X-Requested-With": "XMLHttpRequest"  # Без этого заголовка сервер может вернуть 404
}
```

### 4. Основные эндпоинты
| Операция | Метод | Путь |
|----------|-------|------|
| Логин | `POST` | `/{webBasePath}/login` |
| Список Inbounds | `GET` | `/{webBasePath}/panel/api/inbounds/list` |
| Добавить клиента | `POST` | `/{webBasePath}/panel/api/inbounds/addClient` |
| Удалить клиента | `POST` | `/{webBasePath}/panel/api/inbounds/{id}/delClient/{uuid}` |

### 5. SSL/TLS
- При работе с самоподписанным сертификатом (стандарт для 3X-UI) обязательно:
  ```python
  connector = aiohttp.TCPConnector(ssl=False)
  ```

### 6. Формат данных для addClient
Структура зависит от протокола (автоматически определяется ботом):

**VLESS / VMess:**
```python
{
    "id": 1,  # ID inbound
    "settings": json.dumps({
        "clients": [{
            "id": "uuid-клиента",
            "email": "уникальный-email",
            "limitIp": 1,
            "totalGB": 1024 * 1024 * 1024 * 1024,
            "expiryTime": 1735689600000,
            "enable": True,
            "tgId": "",
            "subId": "hex-строка", 
            "reset": 30,
            "flow": "xtls-rprx-vision" # ТОЛЬКО для VLESS Reality TCP
        }]
    })
}
```

**Trojan:**
```python
{
    "id": 1,
    "settings": json.dumps({
        "clients": [{
            "password": "uuid-как-пароль", # Trojan использует password вместо id
            "email": "...",
            # ... остальные поля идентичны
            "flow": "" 
        }]
    })
}
```

### 7. Описание параметров клиента
| Параметр | Описание | Зачем нам это |
|----------|----------|---------------|
| `id` / `password` | UUID / Пароль | "Пароль" для подключения. <br>VLESS/VMess: поле `id` (UUID). <br>Trojan: поле `password` (UUID). <br>ShadowSocks: для обычных методов UUID или случайная строка, а для методов `2022-*` (напр. `2022-blake3-aes-128-gcm`) **строго** требуются пароли в формате `ServerPassword:ClientPassword` с конвертацией в Base64 (16 или 32 байта, без знаков `=` в конце). Иначе ядро Xray упадет с ошибкой. |
| `email` | Уникальное имя пользователя | В X-UI это поле используется как Name/Label. Должно быть уникальным для Inbound. Мы будем писать туда `user_{id}_{suffix}`. |
| `tgId` | Telegram ID для панели | Позволяет **самой панели** слать уведомления юзеру (через её бота). Нам **не нужно** (у нас свой бот), оставляем пустым. |
| `subId` | Subscription ID | Используется для генерации VLESS-ссылки (как часть пути) и группировки. Мы генерируем его случайно при создании клиента, на стороне панели функционал подписок не используется. |
| `limitIp` | Лимит IP | **Жесткое ограничение по публичному IP.** <br>1 ключ = 1 одновременный IP адрес. |
| `totalGB` | Лимит трафика (Байт) | **Страховочный лимит на месяц.** (Например, 1 ТБ). Включает авто-отключение при превышении. |
| `reset` | Период сброса (дней) | **30**. Включает авто-сброс счетчика трафика на стороне панели каждые 30 дней. |
| `expiryTime` | Срок действия (мс) | **ГЛАВНЫЙ ОГРАНИЧИТЕЛЬ.** Дата окончания подписки. По её истечении панель сама отключит ключ. |
| `flow` | Flow (VLESS Reality) | **Критично для Reality.** Должно быть `xtls-rprx-vision` для TCP+Reality, иначе скорость будет низкой или не будет соединения. Бот определяет это автоматически. |

### 8. Логика смены ключа и лимитов
Пользователь может захотеть сменить ключ (или сервер) в середине месяца.
Чтобы избежать злоупотреблений "сбросом трафика" (пересозданием ключа), вводим ограничение:

1.  **Проверка трафика:**
    Перед сменой ключа бот проверяет текущее потребление (`up + down`) по API.
2.  **Лимит 20%:**
    *   Если потрачено **МЕНЬШЕ 20%** от месячного лимита (200 ГБ из 1 ТБ) -> Разрешаем смену ключа. Старый удаляем, новый создаем (с новым полным лимитом).
    *   Если потрачено **БОЛЬШЕ 20%** -> **ОТКАЗ**.
3.  **Сообщение при отказе:**
    "❌ Вы уже потратили слишком много трафика. Смена ключа будет доступна после сброса лимитов."
    *(Технически получить дату сброса через API сложно (это internal поле), поэтому просто пишем "после сброса/продления").*

- **Продление**: При оплате тарифа или ручном продлении администратором, бот отправляет запрос к API 3X-UI панели на пролонгацию `expiryTime`. Трафик также сбрасывается, чтобы клиент получил свой лимит с нуля.

## 💳 Система оплаты

### Способы оплаты
1. **CryptoProcessing** (https://yadreno.ru/seller) — оплата криптовалютой
2. **Telegram Stars** — встроенная оплата звёздами
3. **Банковские карты (ЮКасса)** — встроенные платежи в рублях через Telegram Payments API

### Тарифы
- Хранятся в БД, управляются администратором
- Каждый тариф: название, длительность (дни), цена в центах, цена в звёздах
- `external_id` — ID тарифа в системе криптопроцессинга

### Механизм pending orders
Для отслеживания платежей используется **предварительное создание ордера**:
1. Пользователь нажимает "Купить ключ" (или продлить).
2. Создаётся запись в `payments` со `status = 'pending'`, уникальным `order_id`. 
   - `tariff_id` и `payment_type` могут быть NULL (заполняются по факту оплаты/выбора).
3. Для крипты: `order_id` передаётся как `invoice` в ссылку.
4. После оплаты — бот получает callback (или успешную оплату Stars), обновляет `tariff_id`/`payment_type` и меняет `status` на `'paid'`. Ключ создаётся или продлевается.

### Формат order_id (invoice)
- **Генерируется из внутреннего ID** записи в таблице `payments`
- **Проверка на внутренний/внешний**: Внутренние ордера (созданные ботом) ВСЕГДА начинаются с префикса `00`. Внешние ордера (пришедшие извне) не должны начинаться с `00`.
- **Кодирование base62**: 0-9, A-Z, a-z — всегда соответствует требованиям криптопроцессинга
- 8 символов base62 = 62⁸ ≈ 218 триллионов комбинаций
- Примеры: `001`, `00a5`, `003D7`

### Callback от криптопроцессинга
Формат: `bill1-ORDER_ID-ITEM_ID-TARIFF-PROMO-PRICE-SIGNATURE`
- `ORDER_ID` — наш invoice_id
- `SIGNATURE` — HMAC-SHA256 подпись (проверяется в [billing.py](bot/services/billing.py))

### Защита от двойного зачисления
- **CryptoProcessing**: проверка `order_id` + подписи при callback
- **Telegram Stars**: `invoice_payload` содержит `order_id`
- Перед зачислением проверяется `status != 'paid'`

### Обработка исключений в оплате
Если после оплаты (crypto callback) выясняется, что оплаченный тариф не найден или неактивен (например, был скрыт/удален администратором):
1. Бот не создает ордер и не выдает ключ.
2. Бот отправляет пользователю специально сообщение:
   > "К сожалению, оплаченный вами тариф не был обнаружен или был удален. Обратитесь в поддержку для уточнения и разъяснения обстоятельств."
3. К сообщению прикрепляется кнопка **«💬 Support»**, ведущая в чат поддержки.
4. Технически это реализовано через исключение `TariffNotFoundError`.

  - `price_cents` — цена в центах ($3.00 = 300)
  - `price_stars` — цена в Telegram Stars
  - `price_rub` — цена в рублях (через ЮКассу), может быть NULL
  - `external_id` — ID тарифа в Ya.Seller (1-9), может быть NULL (используется только в стандартном режиме)
  - `display_order` — порядок отображения
  - `is_active` — активен/скрыт (soft delete)
- **servers** — VPN-серверы 3X-UI:
  - `name` (TEXT) - Название сервера
  - `protocol` (TEXT) - 'http' или 'https'
  - `host` (TEXT) - IP или домен
  - `port` (INTEGER) - Порт панели
  - `web_base_path` (TEXT) - Секретный путь API
  - `login` (TEXT)
  - `password` (TEXT)
  - `is_active` (INTEGER) - 1/0
- **vpn_keys** — ключи/подписки:
  - `user_id`, `tariff_id`, `client_uuid`
  - `server_id`, `panel_inbound_id` — NULL для ненастроенных (черновиков) ключей
  - `custom_name` — пользовательское название
  - `expires_at` — дата истечения
  - `is_enabled` — включён/выключен (1/0), по умолчанию 1
- **payments** — история оплат:
  - `vpn_key_id` — к какому ключу (NULL для нового)
  - `user_id`, `tariff_id` (NULL для pending), `order_id` (уникальный)
  - `payment_type` — 'crypto', 'stars', 'trial' или NULL (до оплаты)
  - `amount_cents`, `amount_stars`, `period_days` (NULL для pending)
  - `status` — 'pending', 'paid', 'expired'
  - `paid_at` — дата оплаты

### Примечания
- **Inbounds** не хранятся в БД — получаются динамически через API панели
- Пользователь может иметь несколько ключей (параллельные подписки)
- **is_banned** = 1 означает что пользователь забанен
- **order_id** в payments — защита от двойного зачисления

## 📅 Ежедневные автоматические задачи

Бот автоматически выполняет следующие задачи каждый день в 03:00 (по локальному времени сервера):

### Суточная статистика (03:00)
Отправляется всем администраторам из `ADMIN_IDS`:
- **Пользователи**: всего / новых за сутки
- **VPN-ключи**: всего / активных / истёкших / созданных за сутки
- **Платежи за сутки**: количество, сумма в USDT и Stars
- **Статус серверов**: онлайн/оффлайн, количество клиентов, трафик, CPU

### Бэкап (03:05)
ZIP-архив с именем `backup_YYYY-MM-DD.zip` отправляется всем администраторам.

**Содержимое архива:**
- `vpn_bot.db` — база данных бота (пользователи, ключи, платежи, настройки)
- `server_НАЗВАНИЕ_x-ui.db` — база каждого активного VPN-сервера (бот опрашивает набор эндпоинтов, таких как `/panel/api/server/getDb`, `/panel/setting/getDb`, для поддержки разных форков панели X-UI/3X-UI)

### Уведомления об истечении (03:10)
Проверка всех активных ключей и отправка уведомлений пользователям, чей срок действия истекает через `notification_days` (настройка в БД, по умолчанию 3 дня).
Текст уведомления также берётся из настроек БД.

**Реализация:** модуль [scheduler.py](bot/services/scheduler.py)
